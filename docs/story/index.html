The Origin Story

How InfantJS
Came to Be

A programming language born from one question: what if Baby.js was still too hard?

01

The Joke

It started as a joke. Our professor introduced a toy language called "Baby.js" as a teaching vehicle for compiler concepts — a stripped-down runtime with simple syntax meant to illustrate how grammars, parsers, and semantic analyzers actually work. We looked at each other and thought: why stop there? If Baby.js exists, there must be something even more foundational. Something even a literal infant could understand. Something so simple, so pure, so wonderfully absurd that it could only have been designed by four sleep-deprived computer science students at two in the morning.

02

The Language

So we built InfantJS — a fully compiled programming language whose entire vocabulary is drawn from the sacred lexicon of babies. You declare variables with mine. You print with gibberish. You loop with wawawa, branch with peekaboo, and return from functions with bedtime. Boolean true is gaagaa. Boolean false is googoo. Random numbers come from flippy. User input is consumed via nomnom. Under all that adorable syntax is a real compiler pipeline: an Ohm-based parser, a static semantic analyzer with type checking and scope resolution, an optimizer, and a code generator that outputs clean, runnable JavaScript — or Python, if you prefer.

03

The Truth

The deeper truth is that InfantJS is a love letter to compiler design. Building a language from scratch forces you to reckon with every assumption you've ever made about programming. What is a type, really? What does scope actually mean? Why do some errors have to be caught at compile time, and not at runtime? We grappled with all of it — and wrapped the answers in the warmest, silliest, most infantile packaging we could imagine. We hope InfantJS makes you smile. We hope it makes you think. And we hope it proves, once and for all, that the best way to learn something serious is to refuse to take it seriously.
